# Discord Scheduled Bot

Бот отправляет сообщения в Discord по расписанию, тегает роль и ставит реакцию `✅`. Управление идет через Telegram-бота.

## Что умеет

- отправка по расписанию в `Europe/Moscow`;
- Telegram-уведомление об успешной отправке по расписанию;
- Telegram-уведомления об ошибках;
- защита от дублей при перезапуске около времени события;
- `/status` для проверки состояния;
- `/reload` для перечитывания `events.json`;
- `/schedule` для управления расписанием кнопками;
- `/send_now` для ручной отправки;
- `/test` для тестовой отправки с ролью `1203431256419995668`.

## Telegram-Команды

Команды доступны только чату из `TG_CHAT_ID`. Если добавить `TG_ADMIN_USER_ID`, бот дополнительно будет проверять Telegram user id отправителя. Это полезно, если команды используются в группе.

```text
/status
```

Показывает онлайн-статус, часовой пояс, количество включенных/отключенных событий и ближайшие отправки.

```text
/schedule
```

Открывает меню с кнопками:

- список событий;
- добавить событие;
- изменить событие;
- включить/отключить событие;
- удалить событие;
- перечитать `events.json`.

Добавление события начинается кнопкой. Затем бот попросит отправить сообщение:

1. название события;
2. текст сообщения;
3. cron-расписание.

Все подсказки будут приходить по очереди. Роль в текст добавлять не нужно, бот добавит ее сам.

Изменение события тоже идет через кнопки:

1. выберите событие;
2. выберите параметр: название, текст или расписание;
3. отправьте новое значение сообщением в Telegram.

```text
/next_events
/next_events 10
```

Показывает ближайшие события.

Формат отображения событий:

```text
• Сб 18:10 — реаки 15x15 ВЗП
```

```text
/send_now
```

Показывает кнопки событий для ручной отправки.

```text
/test
```

Показывает кнопки событий для тестовой отправки. В тесте вместо боевой роли тегается:

```text
1203431256419995668
```

## Формат events.json

```json
{
  "channel_id": 1199512515755909171,
  "role_id": 1199509896069124106,
  "reaction": "✅",
  "allowed_mentions": "roles",
  "events": [
    {
      "name": "weekly_15x15_vzp_saturday",
      "text": "реаки 15x15 ВЗП",
      "cron": "10 18 * * sat",
      "enabled": true
    }
  ]
}
```

`enabled: false` отключает событие без удаления.

На BotHost `events.json` используется как стартовый шаблон. При первом запуске бот создаст рабочий файл расписания:

```text
$DATA_DIR/schedule.json
```

Все изменения через Telegram будут сохраняться туда, поэтому обновление кода из GitHub не должно сбрасывать расписание. Если нужно указать свой путь вручную, добавьте переменную `SCHEDULE_FILE`.

## Переменные Окружения BotHost

```env
DISCORD_TOKEN=ваш_токен_discord_бота
TIMEZONE=Europe/Moscow
EVENTS_FILE=events.json
SCHEDULE_FILE=
LOG_LEVEL=INFO
LOCK_FILE=.bot.lock
TG_BOT_TOKEN=токен_telegram_бота
TG_CHAT_ID=ваш_chat_id
TG_ADMIN_USER_ID=
ENABLE_DISCORD_COMMANDS=false
DISCORD_GUILD_ID=
```

Используйте именно `TG_BOT_TOKEN`, потому что BotHost может иметь системную переменную `TELEGRAM_BOT_TOKEN`.

`TG_ADMIN_USER_ID` необязателен. Если оставить пустым, доступ ограничивается только `TG_CHAT_ID`.

## Структура Кода

- `main.py` — точка входа и обработка аварийного завершения;
- `bot_client.py` — Discord-клиент и slash-команды, если они включены;
- `schedule_service.py` — расписание, перезагрузка событий, ручная и тестовая отправка;
- `telegram_admin.py` — Telegram-команды, кнопки и пошаговое добавление событий;
- `discord_publisher.py` — отправка сообщений в Discord и постановка реакции;
- `stores.py` — чтение/запись расписания и антидубль;
- `formatting.py` — отображение событий в человекочитаемом виде;
- `app_config.py` — переменные окружения и загрузка конфига.

## Деплой На BotHost

1. Подключите репозиторий `xyir4ik/beifonchik`.
2. Ветка: `main`.
3. Python: `3.11`.
4. Команда запуска:

```bash
python main.py
```

5. Добавьте переменные окружения.
6. Пересоберите и запустите бота.

В логах должно быть:

```text
Scheduler started
Telegram commands polling started
```

## Антидубль

Бот хранит время последней успешной отправки в `last_sent.json` внутри `DATA_DIR` или текущей папки. Если событие уже отправлялось в последние 5 минут, повторная отправка по расписанию будет пропущена. Ручные отправки и `/test` антидублем не блокируются.
